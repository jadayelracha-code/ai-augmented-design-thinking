# Solution Space · Prototype

> Goal: make your shortlisted ideas tangible as cheaply and quickly as possible. A prototype isn't a product, it's a question made physical, built to learn, not to ship.

The purpose of a prototype is to be tested and, often, to fail cheaply. The faster and rougher you can make an idea real enough to react to, the faster you learn. Polish is the enemy here; polish makes you attached, and attachment makes you defensive in testing.

> 🔨 Build to learn, not to impress. The cheapest prototype that answers your question is the right one.

---

## Tools & methods in this phase

- **Make ideas tangible** — get out of your head and into something people can see, touch, or click.
- **Low-fidelity prototypes** — paper, sketches, cardboard: fast, cheap, disposable.
- **Virtual prototyping** — digital mockups and simulations.
- **Storyboard** — sketch the user's experience as a sequence of frames.
- **Scenarios** — written walkthroughs of how a user would encounter and use the solution.
- **Videos** — short films showing the concept in use (great for service/experience ideas).
- **Click dummy** — a clickable but non-functional mockup that simulates the real flow.

---

## Where AI genuinely helps

**1. Drafting prototype content instantly**
AI writes the copy, microcopy, and placeholder text that make a mockup feel real, in seconds.

**2. Storyboards and scenarios**
Feed it an idea and get a first-draft storyboard sequence or usage scenario to refine.

**3. Generating variations to test**
Need three versions of a landing page or flow to compare? AI drafts them fast.

**4. Click-dummy scaffolding**
AI can generate the HTML/structure for a simple clickable prototype, accelerating virtual prototyping.

---

## Where AI will mislead you

- **AI makes things look finished too early.** A polished-looking AI mockup can trigger the exact attachment you're trying to avoid. Keep fidelity deliberately low while the idea is still a question.
- It can generate a plausible prototype that quietly drifts from the actual idea or user need. Check it back against the idea napkin and problem statement.
- A prototype's job is to test ONE key assumption. AI happily adds features and polish that blur what you're actually testing. Stay focused on the question.

## Ready-to-use prompts

**Prompt: Draft a storyboard**
```
Idea: [paste idea napkin]. Draft a 6-frame storyboard of a user encountering
and using this solution. For each frame: what's happening, what the user is
thinking/feeling. Keep it simple — this is for sketching, not production.
```

**Prompt: Write a usage scenario**
```
Write a short, concrete scenario of [persona] using [idea] to get their job
done. Ground it in their real pains/gains: [paste]. One or two paragraphs.
Flag anything you're assuming so I can verify it.
```

**Prompt: Generate variations to test**
```
Here's the core concept: [describe]. Give me 3 distinct versions of [the flow /
landing page / message] that make different bets, so I can test which resonates.
Note what assumption each version is testing.
```

**Prompt: Scaffold a click dummy**
```
Create a simple, clickable HTML mockup for this flow: [describe screens and
steps]. Non-functional is fine — just enough to simulate the experience for a
user test. Keep it low-fidelity; this is a prototype, not a product.
```

---

## Before moving to Test, check:

- [ ] Is my prototype cheap and rough enough that I won't be precious about it?
- [ ] Does it clearly test the ONE key assumption I most need to learn about?
- [ ] Did I resist letting AI polish it into something that looks done?
- [ ] Does it still match the actual idea and the user's real problem?
- [ ] Is it real enough that a user can react to it, without over-building?
